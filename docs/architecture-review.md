# OpenStuff 架构与目录合理性评审（v0）

> 目标：在编码前先验证当前目录结构与核心实现逻辑是否合理，并明确建议调整项。

## 1. 当前结构评审结论

整体结构**方向正确**，已经具备从 0 到 1 的可扩展性：
- `apps/macos` 承担 GUI 入口，符合产品定位（macOS 图形应用）。
- `core` 聚合通用能力，便于后续把模式与具体平台解耦。
- `modules` 以业务模式拆分，利于演进与 A/B 测试。
- `scripts` 用于知识解析与 skill 转换，符合“先工具链后产品化”的实施路径。
- `vendors/openclaw` 作为外部依赖隔离目录，便于版本固定和升级。

## 2. 目录结构合理性判断

### 2.1 合理点

1. **按职责分层清晰**
   - UI（`apps`）与核心能力（`core`）分离，防止业务逻辑侵入界面层。
2. **模式与能力解耦**
   - `modules` 表达“业务流程”；`core` 表达“基础能力”，便于复用。
3. **OpenClaw 集成边界明确**
   - vendor 引入源码，而转换逻辑在 `scripts/skills`，符合边界管理。

### 2.2 可优化点（建议在编码前补齐）

1. 增加 `shared/`（或 `core/contracts/`）
   - 用于放置通用类型、事件 schema、错误码、常量，避免 cross-module 循环引用。
2. 增加 `apps/macos/src` 与 `apps/macos/resources` 规划文档
   - 现在只有目录说明，建议提前约定 UI 代码和资源组织方式。
3. 增加 `docs/adr/`（Architecture Decision Records）
   - 用于记录关键决策（采集策略、存储选型、LLM 协议、执行安全机制）。
4. 增加 `data/` 目录规范（仅本地开发）
   - 例如 `data/raw-events`、`data/knowledge`、`data/logs`，用于演示与调试。

## 3. 程序实现逻辑合理性（端到端）

建议固定为以下主链路：

1. **Capture 层（观察）**
   - 采集输入事件 + 应用上下文 + 可选截图锚点。
2. **Knowledge 层（学习）**
   - 标准化事件 -> 切片为任务 -> 生成知识条目。
3. **Orchestrator（决策）**
   - 根据模式调用预测/执行流程并控制权限。
4. **Executor（行动）**
   - 执行 OpenClaw skills 或本地自动化动作。
5. **Storage（记忆）**
   - 持久化原始事件、结构化知识、执行日志。
6. **Review（反馈）**
   - GUI 展示总结，老师确认后反哺知识库。

此逻辑与产品“老师-学生”范式一致，且支持闭环迭代。

## 4. 最小实现边界建议（防止一期失控）

### 一期只做
- 鼠标左键点击采集（不采集键盘内容）。
- 前台应用名 + 窗口标题。
- JSONL 事件落盘。
- 单任务手动触发解析 -> 产出一个 skill。
- 学生模式仅跑白名单动作（打开 app、点击固定坐标）。

### 暂缓
- 通用视觉 UI 识别。
- 大规模多任务并行调度。
- 云端多设备同步。

## 5. 安全与隐私基线（编码前必须定）

1. 默认本地存储，默认不上传原始事件。
2. 所有执行动作需要日志可审计（含触发来源）。
3. 高风险动作（删除、支付、系统设置）默认禁止。
4. 教学/辅助/学生三模式需有明显状态标识和一键停止。
